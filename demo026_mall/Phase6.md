# demo024_mall Phase 6 演进记录 —— 简单支付模拟与订单支付状态流转

> **前置**：demo023_mall 已完成 Phase 5 订单状态机与取消订单。
> **本阶段目标**：实现最简单的支付模拟（10 → 20），打通「下单 → 支付成功」的基础交易闭环。
> **严格限制**：不接真实支付、不做 Redis、不做 MQ、不做发货、不做退款、不新增支付流水表。

---

## 1. 为什么先做「模拟支付」？

在真实电商系统中，支付是整个交易链路的核心环节。但引入真实第三方支付（支付宝、微信、Stripe）会带来：

- 复杂的回调、签名、幂等处理
- 支付单表、支付状态同步
- 分布式事务（支付成功后通知订单系统）

**教学价值递进策略**：
- Phase 6 先用「最简单的方式」完成 10 → 20 状态流转
- 让学生先理解「支付成功后订单状态变更 + 事务边界」
- 为后续真实支付（Phase 7+）铺平道路

## 3. 核心接口

`PUT /api/order/{id}/pay`

**关键设计**：
- 使用**原子条件更新**防止并发重复支付：
  ```sql
  UPDATE `order` SET status = 20 
  WHERE id = ? AND user_id = ? AND status = 10
  ```
- 只有影响行数 = 1 时才认为支付成功
- 支付成功后**不再扣库存**（创建订单时已扣）
- 支付成功后**不再清购物车**（创建订单时已清）

---

## 4. 支付事务边界

```java
@Transactional
public void payOrder(Long userId, Long orderId) {
    // 1. 原子条件更新 status 10 → 20
    int affected = orderMapper.update(... status=10 -> 20);
    if (affected != 1) {
        throw new BusinessException("订单状态已变化，不能支付");
    }
    // 2. 后续可扩展：生成支付记录、通知等（本阶段不做）
}
```

**为什么不扣库存**：
- 库存扣减已在 `createOrder` 中完成（乐观锁）
- 支付只是「确认交易」，不是「占用库存」的新动作

---

## 5. 为什么重复支付必须明确拒绝？

- 业务上：已支付订单不应被重复扣款（即使是模拟）
- 技术上：防止并发请求导致状态不一致
- 教学上：强化「条件更新 + 幂等」概念
