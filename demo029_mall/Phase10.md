# demo028_mall Phase 10 演进记录 —— 模拟退款流程（demo029_mall 的历史基线）

> **前置**：demo027_mall 已完成 Phase 9 模拟第三方支付回调与支付幂等。
> **本阶段目标**：在已支付未发货的订单上实现模拟退款能力（20 → 60），强调退款事务边界、条件更新并发安全、库存恢复一致性。
> 严格限制：本阶段仅支持全额退款 + 已支付(20)未发货订单，不做真实支付渠道、不做退款申请表、不做部分退款、不做已发货退款。

---

## 1. 为什么支付之后要做退款？

在真实电商中，用户支付后可能因为：
- 商品缺货/描述不符
- 商家无法及时发货
- 用户临时取消需求

此时需要支持“已支付但未发货”的退款，而不是等到发货后再处理退货。

**教学价值**：
- 理解支付确认 vs 交易最终性
- 学习退款如何与支付单、订单状态、库存联动
- 掌握退款与发货的并发冲突处理（条件更新）
- 为后续真实退款网关（支付宝/微信退款 API）做铺垫

---

## 2. 为什么本阶段只支持已支付未发货订单退款？

- 状态机清晰：只有 20(已支付) 可退款 → 60(已退款)
- 已发货(30) 退款涉及物流、部分退货、质量问题，复杂度高，本阶段不做
- 禁止对 10/40/50/60 退款，防止非法状态流转
- 简化教学：聚焦“支付后取消交易”的核心场景

---

## 3. 为什么不做部分退款？

- 简化模型：一个订单一个支付单，全额退款
- 库存恢复逻辑简单（按 order_item 全量回滚）
- 避免引入 refund_amount、退款明细表等额外复杂度
- 后续 Phase 可演进为支持部分退款 + 多退款单

---

## 4. 为什么不新增 refund_order 表？

- 最小改动原则：复用 payment_order 表，新增 status=40（已退款）标记
- 支付单天然与退款强关联（同一个支付单只能有一个退款结果）
- 降低表设计和关联查询成本
- 后续如需“退款申请审核”再引入独立 refund_order 表（支持多次申请、审核状态）

---

## 5. 退款事务边界

`refundOrder(LoginUser, orderId)` 必须使用 `@Transactional`。

事务内必须原子完成：
1. 权限 + 状态校验（订单 20 + 支付单 20）
2. 订单条件更新 `status=20 → 60`
3. 支付单条件更新 `status=20 → 40` + 更新时间
4. 查询 order_item 明细
5. 逐个调用 `skuService.restoreStock(skuId, qty)`
6. 任一失败 → 整体回滚（订单状态不变、库存不变、支付单不变）

保证：退款成功 ⇔ 状态正确变更 ⇔ 库存正确恢复

---

## 6. 并发安全：条件更新防重复退款

核心防护：

```sql
-- 订单
UPDATE `order`
SET status = 60
WHERE id = ? AND status = 20;

-- 支付单
UPDATE payment_order
SET status = 40, callback_time = now()
WHERE order_id = ? AND status = 20;
```

- 两个更新都必须影响 1 行，否则立即抛 BusinessException("xxx 状态已变化，不能退款")
- 杜绝“先 select 再 update”带来的 TOCTOU 问题
- 即使两个用户同时对同一订单退款，也只有一个能成功

---

## 7. 与发货并发竞争

可能出现：
- 用户/客服在退款的同时，SELLER 在调用 shipOrder 尝试发货

解决方案：
- 发货使用 `WHERE status=20 SET 30`
- 退款使用 `WHERE status=20 SET 60`
- 数据库保证：两个操作中只有一个能影响 1 行，另一个抛“状态已变化”
- 最终一致性由先成功者决定（先发货则不能退，先退则不能发货）

---

## 8. 权限设计

- Controller 仅负责提取 LoginUser + 401 未登录
- 所有角色判断、订单归属校验在 Service 层完成：
  - BUYER(3)：必须 `order.user_id == operator.userId`
  - ADMIN(1)：无限制
  - SELLER(2)：允许（本阶段简化）
- 符合“权限下沉 Service”的 Phase 7 以来最佳实践

---

## 9. 库存恢复逻辑

退款 = 交易取消：
- 创建订单时已通过 reduceStock 扣减
- 支付只是“确认支付”，不再次扣库存
- 退款时必须全量恢复 order_item 记录的 skuId + quantity
- restoreStock 使用乐观锁，失败抛异常触发回滚
